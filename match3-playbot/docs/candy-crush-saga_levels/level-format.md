# Candy Crush Saga Level Format Reference

每一关的数据是一个 JSON 对象（`all_levels.jsonl` 每行一关）。本文说明各字段的含义。

> **证据等级图例**（适用于本文所有段 ID / item ID / 字段说明）：
> - ✅ **源码确认**：在反编译的 C++ 类名、字符串字面量、调试符号等中找到直接证据
> - 🟡 **数据推测**：通过 22,205 关的 tileMap / `_itemsToOrder` 统计共现得出的高置信度结论（≥80% 匹配）
> - ⚠️ **未确认**：仅有间接线索或仍存在矛盾，需进一步验证
>
> 证据来源：
> - `libcandycrushsaga.so`（84 MB，未 strip 调试符号，含完整 RTTI 类名与源码路径字符串）
> - `jadx_out/sources/com/king/candycrushsaga/`（Java 层很薄，多为 `ccsm_platform` 桥接）
> - `all_levels.jsonl`（22,205 关，本文统计基础）

## 二、tileMap 编码

`tileMap` 是 `boardRows × boardColumns`（默认 9×9）的字符串二维数组。每个格子的值是若干 **3 位数字片段**的拼接，**段之间无固定顺序约定**，但常见的排列规律是：基础地形 → 果冻层 → 可放糖果标记 → 顶层障碍/糖果/spawner。

```
"001004002005026"
 ↓
[001][004][002][005][026]
  ↓    ↓    ↓    ↓    ↓
基础  果冻  可放  Spawner Spawner
地形  layer 糖果  锚点    视觉标记
```

> **关键事实**：
> - `001` 在 22,205 关中出现于约 58 万格，**所有游戏模式的可玩格几乎都有**（包括纯 Order/Drop down 等无果冻关）。它是「基础地形」标记，不代表果冻。
> - `003` 仅在 Jelly 系模式（Light up/Jelly Order/Jelly Drop down/RR Jelly）出现，其他模式 0 出现 → 它和果冻有关。
> - `004` **每格至多出现一次**（不存在 `004004` 这种叠加）；多层果冻通过其他机制（多个 `001` 叠加，或 strength 字段）控制。
> - 证据：22,205 关全量统计中 `004` 在每格出现次数 `{0: 936132, 1: 600783}`，重复次数始终 ≤ 1。

### 2.1 基础地形与果冻段 (✅ 数据确认)

| ID | 含义 | 出现次数 | 出现规律 | 证据 |
|---|---|---|---|---|
| `000` | 空格（不可用） | 261,858 | 棋盘边界 / 空洞 | ✅ 全模式 |
| `001` | **基础格子地形**（"棋盘上有这格"） | 583,100 | 几乎所有可玩格都包含；不一定意味果冻 | ✅ 数据 |
| `002` | **可放糖果标记**（"该格可生成/移动糖果"） | 564,533 | 普通棋盘格，常与 `001` 同时出现 | ✅ 数据 |
| `003` | **特殊果冻变体**（疑似双倍分果冻 / 装饰果冻） | 112,026 | **仅在 Jelly 系模式出现**（Order/Drop down 等模式 0 出现） | 🟡 数据强证据 |

> **多层果冻怎么编码？** 在数据中 `001` 极少数情况下会在同一格重复出现（最多 8 次），疑似用于「多层果冻」编码，但需要更多样本验证。⚠️ 待确认

| ID | 含义 | 出现次数 | 证据 |

### 2.4 多层障碍的 qty 公式

对 N 个连续段 ID 表示的多层障碍（item 28/29/31/32），qty 计算为：

```
qty ≈ Σ_k  k × count(seg_base + k − 1)
```

⚠️ 此公式建立在「连续 N 个段 ID 表示 N 层」的假设上，参见 § 2.3 末尾的「层级 vs 变体」说明。

实际数据匹配率（剔除 spawner 关卡后）接近 100%，**作为统计公式有效**，但**「层数」语义未确认**。建议在 PlayBot 视觉识别落地时通过截图区分。

## 五、收集目标 `_itemsToOrder.item` ↔ tileMap 段 ID

### 5.1 关键概念澄清 ⚠️

> **`_itemsToOrder.item` 的取值与 `libcandycrushsaga.so` 中的 `SEMIDURABLE_*` 枚举不是同一命名空间。**
>
> - **`SEMIDURABLE_*` 枚举**（值 21–32, 64）是 booster/inventory ID。原始枚举见 [5.4 节](#54-semidurable_-枚举源码确认)。
> - **`_itemsToOrder.item`** 在数据中实际取值为 **1–9, 16, 17, 18, 19, 21, 22, 24, 25, 26, 27, 28, 29, 31, 32, 33–42**（共 30 余种）。**部分**取值与 SEMIDURABLE 枚举数字相同，但语义不一定一致。
>
> 区分二者的最稳妥方法是看「该 item 在 tileMap 中是否有对应静态段 ID」：
> - 有 → 它是棋盘元素（蛋糕、糖衣、UFO、椰子轮、Orlock、Snake、骷髅...）
> - 无 → 它是动态生成 / 关卡内 booster（如 SEMIDURABLE_FREE_SWITCH、LOLLIPOP、PAINT_BRUSH 等）

### 5.3 `_itemsToOrder` vs `ingredients` 的区别

- `_itemsToOrder`：Order 系模式的收集目标（甘草漩涡、糖衣、UFO 等）
