一、项目背景与目标

背景

三消游戏（Match-3）及其变体（Merge 2、Merge 3、Blast 等）是全球手游收入最高的品类之一，但行业内缺乏系统性的中文知识沉淀。相关知识高度分散于 GDC 演讲、学术论文、游戏评测、开发者博客、社区讨论等各类英文/中文来源，从业者无法高效获取。

目标

构建一个面向游戏行业从业者的三消游戏垂直知识库，通过 AI 驱动的全网资料收集与编译，将分散知识系统化、结构化，形成可持续维护的行业参考 Wiki。

核心价值

  • 策划：历史案例库、机制设计原则、关卡设计经验
  • 开发：技术选型参考、性能优化经验、已知问题模式
  • 测试：测试用例来源、质量标准参考
  • 市场/BD：竞品分析、市场数据、买量策略、用户研究
  • 产品/运营：元游戏设计、商业化模型、版号合规

    • 二、范围定义

    覆盖品类（分阶段）

    Phase 1 优先收录（核心主题）：

    品类说明代表游戏
    三消经典（Match-3 Classic）消除同色/同形三个及以上棋子Candy Crush Saga、Bejeweled、Jewel Mania
    三消冒险（Puzzle Adventure）三消+叙事/装饰元游戏Gardenscapes、Homescapes、Royal Match
    Phase 1 只收录以上两个子类，这是三消的核心谱系，也是数据最丰富、研究价值最高的品类。

    Phase 2 扩展收录（第二阶段再做）：

    品类说明代表游戏
    Blast 变体点击消除，非滑动匹配Toon Blast、Toy Blast
    Merge 2合并同类两个升级Merge Mansion、Merge Master
    Merge 3合并同类三个升级Triple Match 3D、Merge Island
    其他变体与三消有设计关联的品类根据资料收录情况决定

    内容范围

    收录：

  • 游戏历史与品类演化
  • 核心机制设计（棋盘、特殊棋子、关卡目标、障碍物）
  • 元游戏系统（生命值、进度、活动、社交）
  • 商业化设计（IAP、订阅、广告变现）
  • 市场数据与竞争格局（全球 + 中国市场）
  • 技术实现与引擎选型
  • 美术风格与用户体验
  • 用户获取与增长策略（付费买量、自然流量、ASO、社区）
  • 制作公司与关键人物档案

    • Phase 1 暂不收录（Phase 2 再做）：

  • Blast / Merge 等变体品类
  • 端游/主机三消游戏（除非对手游有直接设计影响）
  • 玩家向攻略内容（目标受众是从业者，非玩家）
  • 对外发布网站（Phase 1 只做内容，不做建站）

    四、每页内容标准模板

    所有页面强制遵守以下结构（由 LLM 在编译时填充）。数据质量是评价页面价值的首要标准，能附上具体数字的内容要优先呈现。

    重要原则： 强制字段只适用于 wiki/ 编译页面，raw/ 原始资料零门槛。LLM compile 时无法填写的字段统一写 待补充：需要[具体来源类型]，禁止留空，但允许"已知待补充"。

    语言策略

  • 存储语言： 中文（正文）+ 英文（专有名词保留原文）
  • 原始资料： 中英文均收录，LLM 编译时翻译为中文正文
  • 专有名词规范： 统一格式 中文名（English Name），如 消除连击（Cascade）
  • 英文资料优先级： GDC 演讲、市场报告、官方说明文档优先收录

  • 每个 /wiki:research 任务对应一个独立的 raw/ 文件，不共享编辑同一篇 wiki 文章

    十三、用户获取专题

    13.1 背景与重要性

    用户获取（User Acquisition / Growth）涵盖一切让玩家发现并下载游戏的手段，包括付费买量、自然流量（ASO/内容/口碑/社区）、KOL 合作等。这是休闲游戏成功的核心杠杆之一。

    以 Royal Match 为例：Dream Games 通过独特的"保护国王"素材风格（付费）、极强的口碑传播（自然）和精准的渠道组合，在竞争极度激烈的三消赛道实现了从 0 到全球 Top 3 的突破——证明增长策略本身就是产品竞争力的组成部分。

    本专题面向增长/买量/市场/产品方向从业者，以可操作的策略分析 + 可验证的数据为核心，付费和自然渠道并重。

    13.2 收录内容框架

    Meta（Facebook/Instagram）主力渠道，流量最大精准定向、LAL 受众、视频/图文/可玩King、Dream Games、Playrix
    Google UAC / PMAX规模化跑量全自动出价，依赖内置 AI 优化全主流厂商

    素材（Creatives）策略

    三消游戏 UA 素材有明显的行业规律：

  • 失败钩子（Fail Hook）：展示玩家"差一步就赢"的紧张感，刺激点击欲望（Royal Match 的"国王被欺负"即此逻辑）
  • 真实玩法展示：直接录制游戏内操作，降低用户预期落差，提升留存
  • 可玩广告（Playable Ads）：让用户在广告内直接体验关卡，适合三消品类，转化率显著高于视频
  • UGC 风格：手持手机录屏、真人出镜，规避"广告感"，TikTok 渠道效果尤佳
  • 情感叙事素材：Gardenscapes 系列以家园改造剧情为钩子，与核心玩法无关但极度高效

    • 素材监测来源：

  • Facebook Ad Library（facebook.com/ads/library）：查看任意 App 在 Meta 平台的全部在投广告
  • TikTok Creative Center（ads.tiktok.com/business/creativecenter）：查看趋势素材和竞品投放
  • DataEye 数眼：国内买量素材监测，覆盖穿山甲/腾讯/快手等渠道

    • 核心指标体系

    指标全称含义三消行业参考值
    CPICost Per Install每安装成本iOS $1.5-5，Android $0.3-1.5（因地区差异大）
    D1 RetentionDay 1 Retention次日留存优秀 ≥ 40%，行业平均 30%
    D7 RetentionDay 7 Retention7 日留存优秀 ≥ 20%，行业平均 12%
    LTVLifetime Value用户生命周期价值依品类和付费率，高质量三消 $2-8/用户
    ROASReturn on Ad Spend广告花费回报率D30 ROAS > 40% 为健康线
    CACCustomer Acquisition Cost获客成本与 CPI 接近，有机流量越多 CAC 越低
    Payback Period回收周期从安装到回收买量成本的时间健康目标 ≤ 6 个月

    Royal Match UA 案例要点（标杆研究）

    Royal Match 是近年三消 UA 的最成功案例，核心经验：

  • 素材定位差异化：绕开"消除"本身，主打"国王遇险/解救"情感叙事，与 Candy Crush 完全不同的视觉语言
  • 高素材消耗量：团队据报每周测试数十组新素材，快速迭代淘汰低效素材
  • iOS 优先策略：优先在 iOS 获取高 LTV 用户，Meta + ASA 双渠道拉满
  • Meta Bidding + SKAN 适配：ATT 隐私变化后快速适配 SKAN，使用聚合测量工具维持归因精度
  • 全球化投放：优先打美/英/德高价值市场，非美市场用于规模扩张降低整体 CPI

    十四、CLAUDE.md 内容规范

    CLAUDE.md（或 AGENTS.md）是 wiki 的"规训层"，Agent 执行任何操作前必须读取。它定义 taxonomy、强制字段、工作流指令和禁止行为。这是 Phase 1 启动的前置条件，必须在第一次 compile 前完成。

    14.1 文章类型 Taxonomy（6 种，强制枚举）

    type用途对应模板强制数据字段
    mechanic游戏机制/设计模式4.1 模板核心数据表格
    title游戏专题页同 4.1，加 key-metrics收入/DAU/留存
    company公司/团队档案同 4.1，加 company-profile收入/规模/成立时间
    market市场分析/竞争格局4.2 模板核心数据摘要表
    growth用户获取/增长策略4.3 模板指标基准表
    history历史/演化时间线自由结构（可用时间轴）关键节点年份
    history 是唯一允许以时间顺序为骨架的类型。其他所有类型禁止用年份/版本号作为小节标题，时间只作为行内标注或 era tag。

    14.2 Frontmatter 强制字段（7 个，适用 wiki/ 所有页面）

    ---
title: ""          # 必填，非空
type: ""           # 必填，枚举值之一
tags: []           # 必填，至少 1 个
data_richness: ""  # 必填，★ 评级（LLM 自评）
status: draft      # 必填，枚举：draft / reviewed / verified
last_updated: ""   # 必填，YYYY-MM-DD
sources: []        # 必填，至少 1 条（可为 {file: "raw/..."} 或 {url: "..."}）
---

    raw/ 目录无此要求，任何格式均可摄入。

    14.3 CLAUDE.md 工作流指令摘要

    ## Workflow Commands

/wiki:ingest <file|url>
  → 将资料存入 raw/，在 log/week-YYYY-WW.md 追加摄入记录
  → 不修改 wiki/ 任何文件

/wiki:compile <raw-file>
  → 从指定 raw 文件编译/更新对应 wiki 页面
  → 填写所有 frontmatter 字段（不可留空，用"待补充"占位）
  → 检查 sources 字段是否引用了该 raw 文件
  → 若目标 wiki 页面已存在，提取互补内容合并，不覆盖

/wiki:research --plan "<主题>"
  → 自主研究指定主题，可并行调用多个子 Agent（每批 ≤ 5）
  → 产出写入对应 wiki 页面，data_richness ≥ ★★★ 才可合入
  → 遇到重复主题，将新产出存入 alt/，等待人工确认合并

/wiki:audit --process-open
  → 读取 audit/ 所有 status:open 的条目
  → 按 severity 排序（error → warning → note）逐一处理
  → 处理完成后更新 status:resolved，填写 resolved_by 字段

/wiki:lint
  → 执行 7 项自动检查（见第八节 Lint 检查项）
  → 输出报告至 log/lint-YYYY-MM-DD.md

    14.4 禁止行为（LLM 必须遵守）

  • ❌ 将 raw/ 原始文件内容直接复制粘贴进 wiki 页面（必须经过编译/提炼）
  • ❌ 核心数据表格留空（必须有内容或"待补充"说明）
  • ❌ 用年份/版本号作为正文小节标题（history 类型除外）
  • ❌ sources 字段为空（至少引用 1 个来源）
  • ❌ 在未确认合并前直接覆盖已有 wiki 页面（应存入 alt/）
  • ❌ 写无来源标注的精确数字（如"留存率 42%"后面必须有来源）

    • 十五、Audit 反馈机制

    人工在阅读 wiki 页面时发现错误、过时内容、待补充数据，通过结构化的 audit 文件记录，Agent 定期批量处理。

    15.1 Audit 文件格式

    文件路径：wiki/audit/{article-slug}--{YYYYMMDD}--{id}.md

    ---
target: wiki/mechanics/special-pieces.md
anchor: "皇家炸弹（Royal Bomb）的爆炸范围为 3×3"
severity: error          # error | warning | note
status: open             # open | resolved
created: 2026-04-20
---

## 问题描述
实际爆炸范围应为 5×5，而非 3×3。

## 修正依据
GDC 2024 Royal Match 技术演讲，第 12 页截图（见 raw/gdc2024-royal-match-tech.pdf）

## resolved_by
<!-- 处理后填写 -->
resolved_at:
resolved_source:         # 修正所依据的来源（文件路径或 URL）
resolved_note:           # 具体改了什么

    15.2 严重程度定义

    Severity含义处理时限
    error数据错误 / 重大事实错误当周清零（Phase 1 门控 G6）
    warning内容过时 / 来源缺失 / 数据需更新当月处理
    note建议补充 / 行文改进按优先级处理

    15.3 Audit 处理指令

    # 查看所有 open audits（按严重程度分组）
/wiki:audit --list

# 处理所有 open audits
/wiki:audit --process-open

# 只处理 error 级别
/wiki:audit --process-open --severity error

    15.4 纠错来源追踪

    Audit 的 resolved_source 字段与文章 sources 字段联动：

  • 若纠错依据是新的 raw 文件，先 /wiki:ingest 该文件，再在 resolved_source 填写其路径
  • 修正完成后，被修正文章的 sources 字段须增加该纠错来源（即修正也有 provenance）

    • 这样每次数据被推翻，都有完整的证据链：原来的数据从哪来 → 被哪个来源推翻 → 新数据是什么。
