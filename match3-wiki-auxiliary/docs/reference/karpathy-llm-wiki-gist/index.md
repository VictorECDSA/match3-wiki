# Karpathy LLM Wiki Gist

> Gist：https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f
> 作者：Andrej Karpathy

## 工具定位

这是 Andrej Karpathy 发布的一篇概念性文章（Gist），描述了一种用 LLM 维护个人知识库的架构模式：将知识存为一组 wiki 页面，而不是传统 RAG（检索增强生成）向量数据库。这不是一个软件库，而是一种**设计模式和思想**，直接催生了 nvk/llm-wiki 等工具。

**核心洞见**：不要存储原始文档在检索时合成，而是用 LLM 持续地将原始素材合成进结构化 wiki，然后直接查询这个 wiki。Wiki 本身就是知识存储。

## 核心模式

### 三层架构

```
第一层：原始素材（只增不改）
  - 网页剪藏、PDF、笔记、视频转录
  - 入库后不修改
  - 所有派生知识的事实依据

第二层：Wiki 页面（LLM 维护）
  - 每个主题一个 Markdown 文件
  - 从相关原始素材中合成而来
  - 随新素材入库增量更新
  - 人可读、可链接、可浏览

第三层：Schema / 配置（CLAUDE.md）
  - 定义主题分类体系
  - 为每个主题指定研究指令
  - 设置质量标准和必要章节
  - 控制智能体行为
```

### 三种核心操作

**入库（Ingest）**
新的原始素材到达时：
1. 解析并切分素材
2. 识别它与哪些 wiki 页面相关
3. 用所有素材（旧 + 新）重新研究那些页面
4. 用新信息更新相关页面

这是**增量知识复利**：每一条新素材都改善现有页面，而不是孤立地待在检索索引里等待被找到。

**查询（Query）**
用户提问时：
1. 在 wiki 页面中搜索相关内容
2. 如果找到：从相关页面综合作答
3. 如果没找到：搜索原始素材，综合作答，可选地创建新 wiki 页面

与 RAG 的关键区别：主要查询对象是已综合的 wiki，而不是原始文档片段。Wiki 已经过人工优化和预合成——查询延迟低，答案质量高于从原始片段中检索。

**健康检查（Lint）**
定期检查：
1. 找到长期没有素材更新的页面（已过时）
2. 找到相互矛盾的页面
3. 找到没有被任何 wiki 页面引用的原始素材（孤立素材）
4. 找到断裂的 wikilink
5. 对照 schema 主题清单报告覆盖盲点

### 为什么这比纯 RAG 更好

| 维度 | RAG（向量库） | LLM Wiki |
|---|---|---|
| 答案质量 | 不稳定（取决于片段检索） | 高（经过合成和整理） |
| 可浏览性 | 无 | 有（Markdown 页面） |
| 人可读 | 无 | 有 |
| 增量更新 | 追加片段 | 重写页面（完整合成） |
| 上下文效率 | 检索片段 | 检索完整页面 |
| 知识复利 | 无（片段保持孤立） | 有（页面整合多个来源） |
| 可扩展性 | 数百万文档 | ~100-200 页（实际上限） |
| 维护成本 | 低（只加片段） | 较高（更新时需重新研究） |

### 规模上限的争议

Karpathy 的文章承认了一个真实局限：这个模式不能扩展到任意规模的知识库。社区讨论的核心在于：

**Token 膨胀**：随着 wiki 增长，重新研究一个页面需要把所有相关原始素材 + 现有 wiki 页面都投入 LLM 上下文。到 100+ 页、数千条原始素材时，成本和时间代价变得可观。

**知识漂移**：早期研究的 wiki 页面在大量新矛盾素材到达后可能已经过时，直到下次全量 `/wiki:compile` 才会更新。与向量库（总是检索最新片段）不同，wiki 可能滞后。

**实际上限**：这个模式对聚焦主题的 50–200 页个人知识库效果很好。对于企业级知识管理（数千页），RAG 或混合方案更实际。

**Karpathy 的立场**：该模式为聚焦领域的深度和质量而优化，不是为无边界主题的广度。对于聚焦 wiki（如三消游戏知识库），这些取舍对 wiki 方式有利。

## 这种思想的重要性

### 知识复利 vs. 知识检索

传统 RAG 积累原始文档，按需检索。每个文档保持孤立——2022 年报告的片段和 2025 年报告的片段在向量库里并排存放，没有任何综合。

Wiki 模式在**入库时就强制综合**。当 2025 年市场报告到达时：
- RAG：把它的片段加入索引。新旧数据共存，未经调和。
- Wiki：更新 `wiki/market/top-grossing.md`，把新数据与 2022 年数据整合，记录变化，更新结论。

Wiki 读者获得整合后的知识。RAG 用户获得检索到的片段，需要自己在脑子里综合。

### CLAUDE.md Schema 作为知识规格说明

Schema 文件不只是配置——它是对知识库应该包含什么内容的**规格说明**。写好一份 `CLAUDE.md` 迫使你：
- 定义哪些主题重要（哪些不重要）
- 为每个主题指定什么样的回答算是好回答
- 设置 LLM 必须满足的质量门

这是有价值的智识工作，让最终 wiki 比无结构的文档堆远更有用。

## 在 match3-wiki 项目中的作用

### 整个架构的思想基础

match3-wiki 项目本质上是 Karpathy 模式在领域规模上的一次实现：

| Karpathy 模式 | match3-wiki 实现 |
|---|---|
| 原始素材 | `raw/` 文件夹（网页剪藏、报告、转录） |
| Wiki 页面 | `wiki/` 文件夹（nvk/llm-wiki 编译产出） |
| Schema / CLAUDE.md | match3-wiki CLAUDE.md（主题分类、质量门） |
| 入库操作 | `/wiki:ingest` 命令 |
| 查询操作 | `/wiki:query` 命令 |
| 健康检查 | `/wiki:lint` 命令 |

### match3-wiki 的 CLAUDE.md 设计要点

Karpathy 的文章强调 schema 设计是这个模式中杠杆最大的工作。对 match3-wiki，CLAUDE.md 应规定：

**主题分类体系**（来自 PRD §三）：
- `history/` — 品类演进、关键里程碑
- `mechanics/` — 核心循环、道具、元游戏、成长系统
- `variants/` — 子品类定义（消除类、合并类等）
- `market/` — 收入基准、头部产品、地区数据
- `growth/` — 买量素材、渠道、CPI 基准
- `production/` — 团队结构、研发节奏、运营活动
- `entities/` — 单个游戏档案（Candy Crush、Royal Match 等）

**各主题的质量门**：
- 市场页面必须包含：收入前 10 产品、地区分布、数据年份
- 机制页面必须包含：游戏案例、玩家心理依据、实现变体
- 实体页面必须包含：开发商、上线日期、收入层级、核心机制、买量策略

**研究指令优先级**：
- 市场数据：Sensor Tower > AppMagic > 行业博客
- 买量素材：TikTok Creative Center > Facebook Ad Library
- 机制：事后复盘 > 开发者分享 > 玩家社区分析

### 实际规模上限管理

考虑到 Karpathy 指出的 ~100-200 页上限，match3-wiki 应该：
- 目标 80-120 个核心 wiki 页面（不求详尽，但求深度）
- 避免创建微型页面（只有 1-2 段的页面）
- 优先创建整合多个维度的综合性页面
- 用实体页面（每个主要游戏一页）锚定具体数据，让主题性页面保持分析视角

这让 wiki 保持在该模式高效运行的范围内，同时以适当深度覆盖三消领域。

## 局限性

- 不适合无边界知识领域（互联网、所有游戏设计）
- 随语料增大，重新研究的成本上升——全量编译变得昂贵
- 知识漂移是真实存在的，需要纪律性地定期运行 `/wiki:lint` 和 `/wiki:compile`
- 写入路径每次更新都需要 LLM API 访问（不支持离线写入）
- 单智能体实现缺少多智能体方式带来的反偏见优势（这正是 nvk/llm-wiki 在基础模式之上增加的东西）
